# 路徑三 —— 產品經理

## 這是寫給誰的

你是 PM、創辦人、策略專員,或公司裡非工程的主管,而公司已經決定要「用 AI」。你不會親手寫訓練 script。你會決定*什麼*要做、*為什麼*做、*為誰*做、*花多少錢*做、*以及要有什麼護欄*。其他的工程團隊會處理。要把你那一份做好,你需要足夠的流利度,才有辦法戳破唬爛、做估算,以及寫出一份不會要求一個物理上不可能的產品的 spec。

## 目標

走完這條路徑之後,你會:

- 把任何一個產品點子對應到「現有技術可以做到 / 做不到 / 這是不確定的中間地帶」。
- 報得出每次查詢、每位使用者的粗略成本區間,而不是只會說「看情況」。
- 跟團隊一起做出有根據的 buy / build / fine-tune 決定。
- 寫出 AI 功能需求,事先就預留 hallucination、延遲、法規、隱私等介面。
- 讀 model card 和 benchmark 表格時不會被淹沒。

## 先備知識

作為使用者有用過 ChatGPT 或 Claude。不需要寫程式。要能夠大大方方地問「這是真的數字還是行銷數字?」。

## 這條路徑

### 第一步 —— 夠用的機制感

把這幾篇掃過去、建立直覺。不需要背下來,只要知道這台機器的*形狀*。

1. [自助餐廳 vs. 牛排館](../01-why-gpu/cpu-vs-gpu.md) —— 10 分鐘。為什麼 LLM 需要 GPU。
2. [Token 是什麼?](../03-llm-inference/token.md) —— 必讀。Token 是*所有東西*的單位 —— 計價、延遲、context 上限、品質。
3. [一個 token 的一生](../03-llm-inference/life-of-a-token.md) —— 10 分鐘。你會突然懂為什麼有 streaming 這東西,以及為什麼 prompt 不是免費的。

**帶走的重點:** Token 對 LLM 的意義,就像 byte 對儲存的意義。價格、上限、功能規格,最終都收斂到 token。

!!! warning "誤解檢查點"

    **「一個 token 就是一個字。」** 聽起來更接近真的的親戚說法:*「字是語言的自然單位。」* 對人類來說,沒錯。但 LLM 在訓練前會先做 tokenizer,常用字會被切成一個 token,不常用字或數字會被切成很多個。`"Hello"` 是一個 token;`"Hsing-Lin"` 可能是四個;一個很長的 URL 可以是二十個。你的定價模型需要假設英文大約是 1.3 token/字,而 code、JSON、非拉丁文字更多很多。用「字數」算出來的預算可以差到 30–200%。

### 第二步 —— 這一代技術實際上能做到什麼

新章節:[能力地圖](../07-strategy/capability-mapping.md)。一份持續更新的實戰手冊:前沿 model 在哪些事情上可靠地做得好、不穩定地做得好、還做不好。裡面有常見產品分類(客服、搜尋、摘要、code、agent)的具體例子。

**帶走的重點:** 誠實的地圖有三區:綠區(可靠)、黃區(搭配護欄和 eval 能動)、紅區(沒有人類在圈內就別出貨)。大多數失敗的 AI 產品,都是把黃區功能當成綠區功能出貨了。

!!! warning "誤解檢查點"

    **「demo 打動我了,產品就會動。」** 聽起來更接近真的的親戚說法:*「能跑的原型等於降低了建置風險。」* 對典型軟體而言,demo 對產品可行性的預測通常相當準,因為簡單情境和困難情境長得差不多。但對 LLM 功能而言,demo 幾乎永遠是*最好的情境* —— 對抗性、長尾、跨語言、重複、隱微曖昧的輸入,才是實際 production 流量的 80%,而這些輸入上的品質可以比 demo 差 30 個百分點。在押注到 demo 之前,先用具代表性的資料跑 eval。

### 第三步 —— 經濟面

新章節:[LLM 產品的經濟學](../07-strategy/economics.md)。你實際上在付的是什麼:per-token 計價、自託管的固定基礎設施、工程時間、eval/ops 成本。三種產品形狀的完整例子:客服機器人、程式助理、agentic workflow。一套 build vs. buy vs. fine-tune 的決策框架。

搭配 [Prefill vs. decode](../03-llm-inference/prefill-vs-decode.md) —— 你應該要懂*為什麼* prompt token 和 output token 成本不同,懂到可以在白板上解釋的程度。

**帶走的重點:** 一個 LLM 功能的 unit economics 其實是*可讀的*,只要你把它拆成 token 進、token 出、per-query overhead。很多「我們不知道這划不划算」其實是「我們根本懶得算」。

!!! warning "誤解檢查點"

    **「我們該自託管省錢。」** 聽起來更接近真的的親戚說法:*「到某個規模,租就會比買貴。」* 對任何運算密集的服務,這終究會成立,但 LLM serving 的*跨越點*比大家想的高。在每天幾百萬 token 以下,一旦你把一位全職 MLOps 工程師、為尖峰準備的 GPU 餘裕、冗餘、on-call 全部算進去,自託管就會虧錢。自託管的理由應該是控制(資料所在地、model 權重、延遲),不是省錢。

!!! warning "誤解檢查點"

    **「自託管才能保護我們的資料。」** 聽起來更接近真的的親戚說法:*「東西在我手上,我才控制得了。」* 目前所有主要 API 供應商都提供 zero-retention 的企業方案,以合約保證你的 prompt 和輸出不會被儲存、記錄、拿去訓練。在幾乎所有情況下,這些合約保證比你在自家強制執行的還要強。自託管應該是因為法規(例如氣隙隔離)或延遲,不是因為模糊的「隱私更好」。

### 第四步 —— 風險、安全、治理

新章節:[風險與治理](../07-strategy/risk-governance.md)。Hallucination、prompt injection、jailbreak、偏見、著作權、EU AI Act 基礎、AI 供應商的 SOC 2、red-teaming。該問供應商什麼、你的內部 AI 政策該寫什麼。

**帶走的重點:** 最大的產品風險不是「model 說了錯的話」—— 而是「model 說了錯的話,然後採取了不可逆的動作」,或是「model 的錯誤輸出附帶的自信,足以讓使用者相信它」。設計時要圍繞可逆性和可追蹤性。

!!! warning "誤解檢查點"

    **「Guardrail(內容過濾)就是安全的解方。」** 聽起來更接近真的的親戚說法:*「對輸入做 sanitization 就解了 security。」* 對傳統 injection 攻擊,input sanitization 確實走得很遠。但對 LLM 而言,guardrail 是一層很淺的保護 —— 它能抓常見的分類,但大多數出貨後的產品事故都是*語意*上的失敗:充滿自信地給出錯的建議、撈到錯的使用者資料、代替使用者採取他從未授權的動作。安全預算應該按這個順序花:eval、log、嚴格的 scope、對高影響動作的人工審核。

### 第五步 —— 好好寫 AI 功能的 spec

好的 AI 功能 spec 包含大多數 PM spec 不會有的東西:

- **Scope 邊界。** model *絕對不可以*嘗試什麼。「不要回答關於 X 的問題。」
- **拒答行為。** 不確定的時候 model 要說什麼。「不清楚時,請反問。」
- **Eval 標準。** 一份範例和預期行為的 golden dataset。(工程會一起幫忙建。)
- **可接受的成本與延遲範圍。** 兩者的 P50、P95。
- **Fallback。** model 壞了、慢了、錯了的時候會發生什麼事。
- **可觀察性。** 哪些東西要 log,誰要審。
- **人類在環的切點。** 哪裡人類可以介入,怎麼浮現出來。

用(開發者路徑裡的)[Evals](../06-building/evals.md) 當你的 eval 詞彙。

**帶走的重點:** AI spec 是*關於 model 行為的法律文件*。在這裡的模糊,請用你對待帳單流程裡模糊的方式去對待它。

### 第六步 —— 怎麼讀 model card 和 benchmark

現代 model card 裡包含:訓練資料摘要、架構、benchmark 分數、評估方法、已知限制、安全資訊。架構掃過去即可。限制要讀。*仔細*讀 benchmark 方法 ——「MMLU 88.2」到底是什麼意思,會因為 5-shot、0-shot 還是 chain-of-thought 而差很多。

對 benchmark 表格,規則是:*一個 benchmark 能預測你的產品的程度,僅限於你的產品長得像那個 benchmark 的程度。* MMLU 分數預測的是知識問答能力。HumanEval 預測的是短、獨立程式題目上的寫 code 能力。這兩個都不能預測*你的*客服機器人。

!!! warning "誤解檢查點"

    **「Model A 在 benchmark X 上贏 Model B,所以我們該用 Model A。」** 聽起來更接近真的的親戚說法:*「排行榜在排誰最強。」* 排行榜排的是*benchmark 表現*,那是你實際任務的一個低保真 proxy。兩個在 MMLU 差一個百分點的 model,在你的工作負載上可能天差地遠。在下決定之前,永遠要用你自己的、具代表性的資料跑 eval —— 包含成本和延遲,這兩個在公開排行榜上幾乎都看不到。

### 第七步 —— 向上、橫向的溝通

你會被要求,有時還在同一場會裡,(a) 向高層把好處吹起來、(b) 跟工程團隊把期待壓下來、(c) 讓法務放心。一些放口袋裡隨時能拿出來用的句子:

- 「這個在 X 上是可靠的。在 Y 上我們需要人類在環。Z 我們現在還不會出貨。」
- 「在這裡換成小一點的 model 可以砍掉 Z% 的成本,品質大概會有這樣的影響 —— 我們會先量再決定。」
- 「我們的 eval 顯示 P50 延遲是 X 秒;大多數使用者感受到的就是這個。P95 會長一些 —— 我們有這個計畫。」
- 「不,那個 fine-tune 解不掉;那是一個 retrieval 的問題。」

這條路徑的詞彙,就是為了讓你能自在地說出這些句子而設計的。

## 誤解總表

| 誤解 | 它從哪裡來 |
|---|---|
| 一個字 = 一個 token | 以人類為中心的語言直覺。 |
| demo = 產品 | 對一般軟體成立;對 LLM 功能不成立。 |
| 自託管省錢 | 規模經濟 —— 但交叉點比你想的高。 |
| 自託管保護資料 | 「東西在我手上」—— 遇到現代企業方案已經過時。 |
| Guardrail 就是安全 | Input sanitization 的類比 —— 罩不住語意層的失敗。 |
| 排行榜能預測產品品質 | Benchmark 是 proxy,不是預測。 |
| 用 fine-tune 加知識 | 把訓練跟知識傳遞混為一談。 |
| Agentic = 不需要人 | Hype vs. 現實。 |

## 檢查點

在你說這條路徑走完之前:

1. 一個團隊提議要做一個會自動寄信給客戶的 agent。按順序描述你會在出貨之前堅持的三道檢查。
2. 你的 CFO 問:「我們這個功能是不是每次查詢都在賠錢?」你要怎麼生出答案?
3. 一家廠商宣稱自家 model「比 GPT-4 好 20%」。在這個數字有任何意義之前,你需要知道什麼?
4. 你的客服機器人 hallucinate,並且向一位客戶引用了一條不存在的退款政策。走過一次 post-mortem —— 找根因,不找戰犯。
5. 用一句話各解釋一次:token、prefill、KV cache、hallucination、RAG、fine-tuning。(對你奶奶練習一次。)

如果你都做得到,你就裝備齊全了。
