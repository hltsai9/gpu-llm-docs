# 學習路徑 —— 選擇你的廚房位置

這份指南一開始是當作一條線性的導覽寫出來的。但不是每個人都會被派到同一個廚房,也不是每個人都需要同一套工具。

一位把 LLM 接進客服系統的後端開發者,在乎的是 token、context、還有成本。一位在 fine-tune 13B 模型的機器學習工程師,在乎的是 gradient accumulation 和 FSDP sharding。一位在評估功能規模的產品經理,在乎的是模型能力、失敗模式、還有每千次查詢多少美元。一位創意總監在乎的是風格與聲音會不會漂掉。一位基礎設施工程師在乎的是,一張 H100 能不能用 4-bit 裝下 70B 模型,還順便服務二十個使用者。

同一個廚房,不同的站。這一頁就是門口的帶位台。

## 每條路徑長什麼樣子

每一條路徑都用同一套骨架,你可以先快速掃過去再決定要不要走:

1. **這是寫給誰的。** 一個具體的角色 —— 你正在打造或決定什麼。
2. **目標。** 走完這條路徑之後,你會有辦法做到、或說明什麼事情。
3. **先備知識。** 我們假設你已經會什麼。這部分故意寫得很短。
4. **路徑本身。** 一串編號的章節,混合既有章節和為這個角色新寫的章節。每一步都會告訴你*為什麼*它在這裡,以及*要帶走什麼*。
5. **誤解 + 它們從哪裡來。** 不是單純列「哪些想法是錯的」。每一個誤解都會附上*為什麼這是一個容易掉進去的陷阱* —— 通常是一個在隔壁領域成立的真命題,搬過來 LLM 就不成立了。
6. **檢查點。** 一些你在進到下一階段之前,應該要答得出來的具體問題。

## 為什麼要特別列出誤解?

大多數技術上的誤解不是隨機冒出來的。它們來自於把一個在某個領域成立的模型,直接套到另一個領域,而其中有一個關鍵細節已經改變了。「參數越多越聰明」在某些脈絡下是合理的粗略規則,在另一些脈絡下就會變成災難性錯誤。「我們需要自己的模型才能擁有自己的資料」對資料庫來說是對的,但對推論 API 通常不是。

如果我們只是告訴你「不要這樣想」,你會點點頭,然後第一次趕時間的時候就又自己推導回 X。所以每一條路徑上的每一個誤解都會附帶它*原本的真命題*。一旦你看到源頭,那個陷阱就失效了。

<div class="analogy" markdown>
誤解通常不是一個笨想法。它是一個放錯廚房的聰明想法。解藥不是羞愧,而是一張更清楚的地圖,告訴你現在站在哪一站。
</div>

## 關於閱讀順序

這些路徑假設你第一次閱讀時,會照列出的順序讀。沒有人攔著你跳著讀 —— 章節大多是可以獨立看的 —— 但後面的步驟有時會借用前面引入的詞彙。當某條路徑說 *「現在回頭再讀一次 [attention](../03-llm-inference/attention.md),用這個框架重看一次」*,它就是字面上的意思:對熟悉的章節再讀一次,你會得到更多。
