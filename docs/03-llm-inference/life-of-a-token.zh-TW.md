# 一個 token 的一生

讓我們跟著一個 token,從出生到出現在螢幕上,走完整段旅程。

你打了 *「法國的首都是」*,然後按 enter。從你的鍵盤到那個答案之間,GPU 叢集裡會發生一連串事件。先用自助餐廳的角度看整段弧線,之後再一段一段放大。

## 步驟 1:Tokenization —— 把訂單切成碎片

Prompt 會從字串被轉成一串 **token ID**。Token 是次詞(sub-word)單位 —— 以上面那句英文為例可能會被切成 `["The", "capital", "of", "France", "is"]`,但實際上更像是 `[464, 3139, 286, 4881, 318]`,也就是在一個 50,000 到 200,000 項目的詞彙表中的整數索引。

<div class="analogy" markdown>
Tokenization 就是服務生把你的訂單(「五分熟,不加洋蔥」)翻成廚房的速記(「#14,-洋蔥」)。從這裡開始,廚房不再用文字思考 —— 它用數字思考。
</div>

這個步驟跑在 CPU 上。很快。它很少影響效能 —— 除非你有超長 prompt,這樣 token 數本身就變成一種負擔。

## 步驟 2:Embedding —— 去架上拿食材

每個 token ID 會透過在一個 **embedding 表**上的查表,變成一個向量(現代模型常見 4,096 到 12,288 維)。這張表每一列對應到詞彙表裡的一個項目。這個矩陣乘法把離散的「字」映射到連續向量,後面的模型才能對它做算術。

現在你手上有一個形狀是 `(prompt_length, hidden_dim)` 的矩陣。如果是 5 個 token 的 prompt 搭配 4,096 維的 embedding,那就是一個 5×4,096 的浮點數矩陣。

## 步驟 3:Transformer 堆疊 —— 生產線本體

這個矩陣現在會通過 **N 層 Transformer** —— 視模型大小 32、48、80、96 層都有。每一層大致上做一樣的事:

1. **Layer norm。** 對 activation 做正規化(便宜的 elementwise 運算)。
2. **Attention。** 每個 token 看過前面的所有 token,決定要「關注」哪些。這就是那句有名的「Attention is all you need」實際發生的地方。[完整章節](attention.md)。
3. **殘差連接(Residual connection)。** 把 attention 的輸出加回輸入。
4. **Layer norm。** 再正規化一次。
5. **前饋層(Feedforward / MLP)。** 兩層的 MLP,對每個 token 的表徵獨立做轉換。中間隱藏維度通常是 4 倍大。
6. **殘差連接。** 把 MLP 的輸出加回去。

這些運算幾乎每一個都是矩陣乘法(或好幾個),對象是住在 HBM 裡的權重張量。一個 70B 參數的模型,權重會橫跨整個 HBM —— 如果是 80 層的模型,每一層大約有 9 億個參數的權重矩陣。

<div class="analogy" markdown>
每一層都是生產線上的一個工作站。Activation 矩陣是被加工的半成品。在每一站,權重(工具和零件)從裝卸月台(HBM)被搬來,套用到半成品上,之後半成品再往下一站移動。經過 80 站之後,成品完成。
</div>

## 步驟 4:LM Head —— 挑下一個字

走完最後一層之後,每個 token 的向量會經過 **language modeling head** —— 一個大矩陣乘法,把 4,096 維的向量重新投影成一個長度 50,000 的向量,每個詞彙表項目一個 logit。再對這個向量做 softmax,就得到一個機率分佈。

我們只在乎**最後一個**token 的 logits,因為我們要預測「prompt 之後接什麼」。所以挑一個字 —— greedy、top-k、top-p、temperature、nucleus,隨你想怎麼挑 —— 然後下一個 token 就蹦出來了:`「巴黎」`。

## 步驟 5:串流 —— 自回歸(Autoregression)

接下來是招牌技巧。我們「不會」把一切丟掉重來,而是**把新 token 追加到序列末端**、再把整個模型跑一遍,往前走一步,得到再下一個 token。然後再追加,再跑、再追加,直到模型吐出一個結束 token,或是打到長度上限為止。

這叫做**自回歸生成(autoregressive generation)**。每產生一個新 token,都是一次完整的前向傳遞,條件是前面的所有 token。

效能故事從這裡開始有趣起來。如果每產生一個新 token 都要對「整段歷史」重算一次 attention,每個 token 的成本會隨著回答越長而爆炸性增加。為了避免這件事,LLM 用了一個聰明的捷徑 —— **KV 快取**,把 attention 對「前面 token」需要知道的資訊記起來,這樣每個新 token 的前向傳遞只要處理自己加上查一下快取就好。

KV 快取我們會有一整章講它,因為它非常核心。現在先在腦中留下這個畫面:

<div class="analogy" markdown>
廚房在工作時會一邊翻一本筆記本。每做好一道菜,主廚就在筆記本上簡單寫幾句。下一道菜進來時,主廚翻筆記本而不是把所有東西重頭再煮一遍。沒有筆記本的話,每做一道新菜都會比上一道更花時間;有筆記本,每道新菜的工夫就大致固定。
</div>

## 兩個個性截然不同的階段

你應該已經隱約感覺到推論有兩個階段了:

**Prefill。** 第一次前向傳遞會一次平行處理「整個」prompt。工作量很多,但平行度也超高 —— GPU 很開心、Tensor Core 忙得不得了,屬於受限於運算(compute-bound)。

**Decode。** 後續 token 一個一個生成出來。就算有 KV 快取,每產生一個新 token 還是要把整份模型權重從 HBM 拉一次,而對應的算術量卻很少。每一步的平行度極低。受限於記憶體。GPU *不開心*。

[Prefill vs. Decode](prefill-vs-decode.md) 那一章會深入討論。它是「LLM 推論表現」最重要的一刀切分。

## 你實際看到的是什麼

現代聊天介面會看到 token 一個一個冒出來。這是真的 —— 你看到的每一個 token,對應的都是一次完整通過整個網路的前向傳遞。如果伺服器每秒為你生出 100 個 token,就代表它每秒完成 100 次前向傳遞,每一次都會碰到大約 140 GB 的權重(70B 模型在 FP16 下)。

也就是說,光是「你一個」請求,伺服器就要以約 14 TB/s 的速度把資料從 HBM 搬過去。H100 的 HBM 頻寬大約 3.35 TB/s,這代表要 4 到 5 張 H100 全力運轉 —— 只為你一個人服務。這就是為什麼 LLM 服務系統必須聰明:你沒辦法為每個使用者分配 5 張 GPU。系統得把你的請求和另外幾十個請求 batch 起來,讓那些 HBM 讀取被攤分掉。[完整章節:Batching](batching.md)。

## 一行總結整張圖

- **Token 進來 →** embedding 表查詢
- **→ N 層 Transformer →** 每一層做 attention(用 KV 快取)和前饋
- **→ LM head →** 挑下一個 token
- **→ 追加到序列 →** 重複

……GPU 會在「開心(prefill)」與「不開心(decode)」之間切換,而服務系統會拼命把所有請求 batch 起來,盡量不讓 HBM 閒下來。
