# 受限於記憶體 vs. 受限於運算

最後,我們用一個把所有事情串起來的心智模型收尾:**Roofline(屋頂線)**。

Roofline 是效能工程師發明的圖表,用來回答一個很簡單的問題:*在這張硬體上,這個運算最快能跑多快?* 只要把 roofline 內化了,LLM 推論裡幾乎每一個最佳化故事都會立刻對上號。

## 兩個天花板

GPU 上任何一個運算,受限於以下兩件事之一:

1. **運算吞吐量(Compute throughput)。** Tensor Core(或 CUDA Core)做算術的速度有多快,單位是 FLOPs/秒。
2. **記憶體頻寬(Memory bandwidth)。** 資料從 HBM 搬到晶片上的速度有多快,單位是 bytes/秒。

一個運算實際能跑多快,由「哪一邊先用光」決定。你不可能跑得比 Tensor Core 快,也不可能跑得比記憶體通道快。

<div class="analogy" markdown>
工廠的吞吐量由較慢的那一邊決定:機器工作得多快,或原料從裝卸月台送進來的多快。如果堆高機已經開到極限,雇再快的機器也沒用;如果機器已經被餵飽,買更快的堆高機也沒用。
</div>

## 算術密度:x 軸上的那個量

決定你會撞到「哪一個」天花板的量,叫做**算術密度(arithmetic intensity)**:

$$
I = \frac{\text{FLOPs}}{\text{從 HBM 搬來的 bytes}}
$$

就是「每載入一個 byte 資料做幾次算術」。高算術密度 = 每載入的 byte 都被重複使用很多次。低算術密度 = 載一個 byte、做一次乘法、就扔掉。

以 H100 為例(大致上):

- 巔峰運算吞吐量(BF16 Tensor Core):約 989 TFLOPs
- 巔峰 HBM 頻寬:約 3.35 TB/s
- **交界點:每 byte 約 295 次運算**

如果你的運算算術密度低於 295,你就是**受限於記憶體**:HBM 頻寬是天花板,假如資料能更快送到,在原理上你能把全部運算量用滿。如果高於 295,你就是**受限於運算**:Tensor Core 是天花板,就算還有記憶體頻寬可以用,你也不需要用到。

## Roofline 圖本身

把算術密度畫在 x 軸上(對數尺度),能達到的 FLOPs 畫在 y 軸上(對數尺度)。兩條線:

- 一條代表記憶體頻寬的斜線:`y = 頻寬 × x`。算術密度加倍,你能達到的效能就加倍,直到……
- 一條代表運算的水平線:`y = 巔峰 FLOPs`。過了這條線之後,再高的算術密度也沒用。

兩條線在交界點相遇。交界點左邊的運算住在斜線上(受限於記憶體),右邊的運算住在水平線上(受限於運算)。

```
  可達 FLOPs
        │
   peak ├───────────────────  ← 運算天花板
        │                    ╱
        │                   ╱
        │                  ╱
        │                 ╱     ← 記憶體頻寬線
        │                ╱      (斜率 = HBM 頻寬)
        │               ╱
        │              ╱
        └─────────────┴────────── 算術密度
                      ↑
                  交界點(H100 約 295 ops/byte)
```

各種運算就是這張圖上的一個個點:

| 運算 | 算術密度 | 落在哪裡 |
|---|---|---|
| Elementwise 加 / 乘 | 每 byte 約 1 次運算 | 極度受限於記憶體 |
| Layer norm | 每 byte 約 3 次運算 | 受限於記憶體 |
| Matmul,batch=1 decode | 每 byte 約 1–2 次運算 | 受限於記憶體 |
| Matmul,batch=64 decode | 每 byte 約 30–50 次運算 | 受限於記憶體,但靠近一點了 |
| Matmul,prefill(seq=2048) | 每 byte 約 500+ 次運算 | 受限於運算 |
| 大型訓練 matmul | 每 byte 約 1000+ 次運算 | 受限於運算 |
| Attention(天真實作) | 不一定 | 沒有 FlashAttention 時受限於記憶體 |
| Attention(FlashAttention) | 高很多 | 受限於運算 |

## 為什麼 decode 這麼痛苦

走一遍 70B 模型、batch=1 的一次 decode 步:

- 搬的權重:每步約 140 GB。
- 算術量:每個 token 約 140 GFLOPs(大致一個參數一次運算)。
- 算術密度:每 byte 約 1 次運算。

我們在一個需要每 byte 295 次運算才能跑滿的晶片上,只做到 1。也就是說,decode 能用到的 Tensor Core 頂多是約 1/295。其他部分全都在閒著等 HBM 那台堆高機。

整個服務系統其實是花錢買了一張 989 TFLOP 的加速器,結果只讓它跑出約 3 TFLOPs 的有效吞吐量。**這就是 LLM 推論為什麼這麼貴最根本的原因。**

## 什麼能讓你在 roofline 上移動

本指南裡你看過的每個最佳化,都可以讀成 roofline 上的一個移動:

**Batching** —— 向右移(更高的算術密度),因為把 HBM 讀取分攤到更多工作上。把 decode 從「受限於記憶體的最左邊」推到「靠近受限於運算」的地方。當 KV 快取開始主導流量時,邊際效益就會遞減。

**量化** —— 直接**把 byte 縮小**,所以也是往右移。如果權重從 FP16 改成 FP8,同樣的運算次數 HBM 流量只剩一半,所以算術密度翻倍。同時也會把整條頻寬線往上推一點點(等效每秒更多 byte)。

**FlashAttention** —— 因為避免把中間分數矩陣寫回 HBM,所以把 attention 往右推。在不改變數學的情況下,把一個受限於記憶體的運算轉成受限於運算。

**運算融合(Operator fusion)** —— 把多個 elementwise 運算合併成一個 kernel,中間值只在暫存器裡流動,不再往返 HBM。融合後的運算算術密度會上升。

**KV 快取的各種招式(GQA、paging、量化)** —— 減少每次 decode 要讀的 byte,所以算術密度上升。

**硬體升級** —— H200 比 H100 多約 1.4 倍的 HBM 頻寬。如果你是受限於記憶體,那差不多就是多 1.4 倍的 token/秒。Blackwell 加的是更多 FLOPs;對受限於運算的工作(prefill、訓練)有幫助。

<div class="analogy" markdown>
LLM 推論的每一個小聰明,本質上都是兩種動作之一:「讓堆高機更快」(硬體、記憶體頻寬)、或「讓每趟堆高機路程運回更多有用的工作」(batching、量化、融合、聰明的快取)。沒有人浪費時間多雇廚師 —— 廚師早就在那邊閒著。
</div>

## 遇到新技術時怎麼推理

下次看到新的 LLM 效能優化論文或部落格,你可以拿這份清單過一遍:

1. **這個運算的算術密度是多少?**(FLOPs / byte。)
2. **它現在是受限於記憶體還是受限於運算?**(算術密度是在交界點之下還是之上?)
3. **這個技術改變了什麼 —— FLOPs、bytes、還是兩者?** 通常是減少 bytes。
4. **這樣會把運算推到更高使用率的區域嗎?**

如果這個技術在一個「受限於記憶體」的運算上減少了 bytes,加速比大致會是成比例的。如果在「已經受限於運算」的運算上減少 bytes,幫助就小了。如果在一個「受限於記憶體」的運算上減少 FLOPs,甚至可能讓事情更慢(沒幫到瓶頸,反而增加複雜度)。

這副眼鏡能幫你過濾掉很多行銷話術。

## 為什麼 roofline 會漂移

最後一個觀察。交界點在一代一代之間**不斷往右移**:

- Pascal(2016):每 byte 約 15 次運算才能吃滿。
- Volta(2017):每 byte 約 70 次。
- Ampere(2020):每 byte 約 140 次。
- Hopper(2022):每 byte 約 295 次。
- Blackwell(2024):更高,尤其在 FP8/FP4。

每一代之間,運算吞吐量的成長都比記憶體頻寬更快。每張新 GPU 都變得更「頭重」。也就是說,*同樣*一份工作負載在 Pascal 上可能是受限於運算,在 Hopper 上就變成受限於記憶體了。演算法沒變,但底下硬體的形狀變了。

這就是為什麼 LLM 服務系統這幾年越來越複雜、而不是越來越簡單。最早的推論引擎只是把 matmul 丟進 cuBLAS 就交差了事;現代的則在記憶體階層上動刀 —— 因為硬體失衡的地方就在那邊,逼著你非面對不可。

## 用三句話總結整本指南

- GPU 長得像自助餐廳:很多廚師,在不同食材上做一樣的事。
- LLM 長得像一個體育場觀眾群的訂單:一大堆完全一樣、彼此獨立的算術。
- 有意思的效能故事,從來不是在講廚師 —— 而是在講「幫廚師送食材的那些堆高機」,以及這整個領域所有的最佳化,都是在想辦法讓每一趟堆高機之旅送出更多有用的工作。

你已經有了心智模型。規格書上的數字、服務系統部落格文章的標題、推論設定檔裡的旋鈕,應該都能讀成「硬體 × 工作負載」這張契合度表上合理的結果。歡迎來到自助餐廳。
