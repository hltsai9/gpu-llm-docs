# Token 是什麼?

在跟著它走完整段旅程之前,我們應該先靠近一點看看它。

*Token* 這個詞被隨口亂用的程度,跟「字」差不多。它們不是同一回事。Token 是一段文字 —— 有時候是一整個字、有時候是一個片段、有時候是單一個位元組(byte)—— 並且配上一個整數 ID。LLM 做的每一件事,單位都是 token:prompt 用 token 計算、上下文長度用 token 衡量、價格用 token 報價、吞吐量也用每秒幾個 token 回報。

如果後面每一章都會繼續用這個字,那值得花一章先問清楚:它到底是什麼東西?

## Token 就是一個整數

從模型的角度看,*「法國的首都是」*這句話不是中文,也不是英文。它比較像這樣:

```
[464, 3139, 286, 4881, 318]
```

五個整數。每一個對應到一張詞彙表的某一列,而這張表可能有 50,000 到 200,000 個項目。這張表是「訓練時」就一次決定好的,模型的一生都不會再改。

<div class="analogy" markdown>
想像一家有 50,000 個編號餐點的餐廳。你不會點*「黑麥麵包夾火腿起司加芥末」*,你會點 *#14,加芥末*。廚房不看文字,它看數字。Tokenization 就是進門時幫你翻譯的服務生,detokenization 則是最後在收銀台上把收據印出來的那個動作 —— 用同一張表反過來查回去。
</div>

模型本身 —— 每一次矩陣乘法、每一個 attention head、每一層 —— 運作的對象都是那些整數(嚴格來說是整數查表查出來的向量)。你在螢幕上看到的文字,是最後一步才透過同一張表「反拼」出來的。

## 為什麼不直接用「字」當作 token?

很直覺的想法是:一個 token 等於一個字。這個做法很快就會破功,有幾個理由。

**沒見過的字會整個卡住。** 一個在 2018 年訓練的「字」級 tokenizer,完全沒有 *ChatGPT*、*rizz*、*Llama-3* 這些項目。你必須替「從來沒看過的字」想好怎麼辦。

**詞彙表會爆炸。** 英文有上百種變化形(*run, runs, ran, running, runner*),而芬蘭文、土耳其文這類「黏著語」甚至有上百萬種詞形。Embedding 表和 LM head 的大小會隨詞彙量線性增加,一張百萬列的表在記憶體和輸出層運算上都會慘到無法接受。

**名字、程式碼、網址都不是「字」。** `s3://my-bucket/logs-2024-04-17.parquet` 不是一個字,`np.linalg.svd` 也不是,`x0bf3c2e1` 更不是。能用的 tokenizer 必須優雅處理它們。

## 那用「字元」呢?

另一個極端:一個字元一個 token。這樣確實解決了詞彙量和未知詞的問題 —— 整個 Unicode 大概只有 15 萬個碼位,沒有什麼東西會「超出詞彙表」。但這對效率來說是災難。

一個字元級的模型會把 attention 的預算花在學「t-h-e 連在一起」這種事上面,才能學到任何有趣的東西。序列長度會膨脹 4 到 5 倍,而 Transformer 裡每一個運算都跟序列長度有關 —— attention 本身就是 $O(n^2)$ 的。訓練和推論都會變慢 4 到 5 倍,這代價太高,沒有任何量產的 LLM 會付。

## Sub-word(次詞):折衷做法

現代的 tokenizer 採取中間路線:**次詞單位**。長度夠長,所以效率還不錯;短到任何字串都能用它們拼出來。

常見字保持完整:`" the"`、`" and"`、`" is"`。罕見字會被切開:`"antidisestablishmentarianism"` 可能會變成 `["anti", "dis", "establish", "ment", "arian", "ism"]`。任何奇怪的東西 —— 人名、程式碼、emoji、隨機位元組 —— 永遠可以一路退回更小的片段。

這給你:

- **詞彙表大小可控。** 30K 到 200K 項目,不是幾百萬。
- **沒有未知詞。** 退路永遠存在,因為 tokenizer 認得每一個位元組。
- **序列長度合理。** 英文通常是每個單字 1.2 到 1.5 個 token。

建這種詞彙表最常見的演算法家族叫做 **Byte-Pair Encoding(BPE)**。

## BPE 如何建出詞彙表

BPE 從「最小的字母表」開始 —— 位元組,或 Unicode 字元 —— 然後**貪婪地合併**。

1. 以每一個位元組(256 個)或語料裡出現過的每一個 Unicode 字元當作初始詞彙。
2. 計算「每一對相鄰符號」在語料裡出現的次數。
3. 把最常出現的那一對合併成一個新的詞彙項目。
4. 把語料裡所有這一對都換成新符號,回到第 2 步。
5. 當詞彙量達到目標(例如 50,000 個)就停止。

每一次合併都會「照順序」記下來。推論時,tokenizer 對你的輸入重播同一串合併:從位元組開始,依訓練時的合併順序,能合就合。合完之後出來的就是 tokenization 的結果。

<div class="analogy" markdown>
想像你讀過一大堆語料,然後觀察到:*t 跟 h 一天到晚黏在一起,我乾脆不要再把它們當兩個字母,而是把 `th` 當一個東西。* 接著:*`th` 跟 `e` 也很常一起出現,那把 `the` 也當一個。* 然後:*空白加 `the`。* 做完上萬次這種貪婪合併,英文裡常見的片段就變成單一的 token,只有比較罕見的東西會留在被切開的狀態。
</div>

這就是為什麼在 GPT 家族的 tokenizer 裡,你會看到像 `" the"`(開頭帶空白)這樣的單一 token。空白是 token 的一部分,因為在真實文字裡它的確「大多數時候」都和 *the* 黏在一起。

## Byte-level BPE:一招走天下

GPT-2 引入了一個小但後果重大的變化:BPE 跑在**位元組**上,不是字元上。起始字母表剛好 256 個符號(位元組值 0 到 255)。地球上任何字串 —— 中文、英文、emoji、二進位資料、格式壞掉的 Unicode —— 都是位元組序列,所以**永遠不會發生詞彙表外的情況**。最糟的情況,一段奇怪的輸入會被逐位元組切開,每個位元組變成一個 token。

這就是為什麼你可以把任何東西貼進一個現代 LLM,它至少都能被 tokenize。Tokenizer 不會當掉。唯一的代價是不熟悉的文字會吃掉很多 token。

!!! note "Unicode、位元組,以及為什麼這件事重要"
    在 UTF-8 下,一個中文字通常是 3 個位元組。如果 tokenizer 在訓練時完全沒看過中文,每個字可能就吃 3 個 token。如果看過,常見字和詞會被合併成 1 到 2 個 token。這件事會直接反映在你的帳單上:同一句話,用舊的 tokenizer 可能中文比英文貴 2 到 3 倍的 token,用新的多語系 tokenizer 大約是 1.2 到 1.5 倍。

## 詞彙量 —— 一個取捨

那為什麼不乾脆用「更大」的詞彙表,讓所有東西都變成一個 token?

**輸出層會變貴。** LM head 把 hidden dim(~4,096)投影到詞彙量(~50,000)。詞彙量翻倍,這一層的參數和每次前向傳遞的運算也會翻倍。

**Embedding 表會變大。** 4,096 × 200,000 × 2 bytes(FP16)就是 1.6 GB,光是輸入 / 輸出的 embedding 就這麼大(兩者常常共用同一張表)。

**邊際效益遞減。** 超過 10 萬項之後,每新增一個 token,能擷取到的「片段出現頻率」都越來越低。你花了一堆參數,只為了把平均序列長度再砍掉一點點。

現代 LLM 落在這幾個區間:32K(Llama 2)、128K(Llama 3、GPT-4 turbo)、200K(Claude、GPT-4o)。整體趨勢往上,主要是因為多語系覆蓋會從更大詞彙表中明顯受益。

## 特殊 token(Special tokens)

不是每一個 token 都是英文或中文的片段。Tokenizer 會保留一小部分**特殊 token**,它們是講給系統聽的,不是講給讀者聽的。

| Token | 意義 |
|---|---|
| `<bos>` / `<s>` | 序列開始(beginning of sequence)。 |
| `<eos>` / `</s>` | 序列結束。模型吐出這一個就代表「我講完了」。 |
| `<pad>` | 填充(padding),用來把 batch 裡的序列長度對齊。 |
| `<unk>` | 未知(unknown)。byte-level BPE 裡很罕見,但舊的 tokenizer 會用到。 |
| `<|system|>`、`<|user|>`、`<|assistant|>` | 聊天模型裡的角色標記,用來分隔系統提示、使用者回合、以及模型自己的輸出。 |

特殊 token 跟一般 token 共用同一張詞彙表,也有正常的整數 ID。但它們不會出現在使用者看到的文字裡 —— detokenization 時會被拿掉,或由聊天介面另外處理顯示。

<div class="analogy" markdown>
特殊 token 就像劇本裡的舞台指示。它們寫在劇本裡,好讓演員(模型)知道一場戲什麼時候開始、什麼時候結束,但觀眾(使用者)從來不會真的「聽到」這些話。如果你組 prompt 時忘了放對的角色標記,模型會搞不清楚現在輪到誰講話 —— 就像演員拿到一份沒有換場提示的劇本,演起來會錯亂一樣。
</div>

## 為什麼中文比較貴

Sub-word tokenizer 是在語料上訓練出來的。哪種語言在語料裡佔比重,它就會被分配到有效率的 token;其他語言就得多付錢。

以 OpenAI 比較舊的 `cl100k_base` tokenizer 來說,每單位資訊要吃的 token 大致長這樣:

- 英文:每個字 ~1.3 個 token、每個 token ~4 個字元。
- 西班牙文 / 法文 / 德文:~1.5 到 2 倍英文。
- 中文 / 日文 / 韓文:~2 到 3 倍英文。
- 泰文 / 阿拉伯文 / 印地文:有時候到 3 到 4 倍英文。

新一點的 tokenizer(GPT-4o 的 `o200k_base`、Llama 3 的 128K 詞彙、Claude 的 200K)把差距大幅縮小,但沒有消除。實務上的後果:一個 API 呼叫拿英文段落做摘要花 $0.01,中文同樣長度的段落可能變成 $0.02 到 $0.03,而且上下文長度也會更快被吃滿。

!!! tip "怎麼查你的 tokenizer"
    OpenAI 有公開 `tiktoken`(`pip install tiktoken`),可以針對 GPT 模型精確算 token 數。Hugging Face 的 `AutoTokenizer` 則對開源模型可以做一樣的事。如果只要粗估:一個英文單字 ≈ 1.3 個 token,在新的 tokenizer 下一個中文字 ≈ 1 到 2 個 token。

## 這件事對 GPU 的意義

整個效能故事,都是用 token 當單位在講的。

- **吞吐量。** 服務端 benchmark 回報的是「每秒每 GPU 幾個 token」(TPS),這是最根本的貨幣。
- **延遲。** 首個 token 的時間(TTFT)和 token 間延遲(ITL)都以 token 為單位衡量。
- **上下文。** 128K context window 指的是模型一次能看到 128,000 個 token —— 不是字元、不是字、是 token。
- **價格。** API 定價字面上就是「每百萬個輸入 token 多少錢、每百萬個輸出 token 多少錢」。

後面每一章都假設你知道 `[464, 3139, 286, 4881, 318]` 裡那個整數*代表什麼*。現在你知道了。

## 一章的重點,四行收完

- 一個 token 是一個整數 ID,指向一段次詞文字。詞彙量通常 30K 到 200K。
- 次詞 tokenization(通常是 BPE)是「字級(太大、太脆弱)」和「字元級(太慢)」之間的折衷方案。
- Byte-level BPE 保證任何輸入都能被 tokenize。永遠不會「超出詞彙表」。
- 不是每種語言都被同等對待 —— 非英文的文字通常要花更多 token 才能表達一樣多的意思,這會直接反映在「回答變慢」和「帳單變貴」兩件事上。
