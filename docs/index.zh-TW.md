## 這份指南寫給誰看

這是一份循序漸進的指南。前面的章節只假設你會寫程式,其他背景知識都不太需要。後面的章節會更深入地討論記憶體頻寬、算術密度,以及現代 LLM 推論引擎如何榨乾每一個時脈週期的效能。第一次閱讀請按順序讀;第二次再當成參考書使用。

## 核心問題

現代的 LLM 本質上就是一大堆乘法運算堆疊起來的結果。70B 參數模型的一次前向傳遞,大約會觸碰到 140 GB 的權重、執行數十兆次運算 —— 而且這還只是為了產生「一個」token。把這件事每秒重複好幾次、同時服務成千上萬個使用者,你就得到 GPU 當初被設計來處理的那種工作負載了。

但「GPU 很快」並不算是一個解釋。GPU 的快,是非常特定的一種快。它在很多 CPU 擅長的事情上表現糟糕,但對 Transformer 需要做的事情卻非常拿手。要理解「為什麼」,我們需要先建立幾個基本概念:

<div class="numbers" markdown>
<div class="cell"><div class="n">約 10,000 個</div><div class="l">現代資料中心 GPU 上的 CUDA 核心數量</div></div>
<div class="cell"><div class="n">約 3 TB/s</div><div class="l">H100 的 HBM 記憶體頻寬</div></div>
<div class="cell"><div class="n">約 1,000 倍</div><div class="l">GPU 與 CPU 在矩陣乘法上的吞吐量比</div></div>
<div class="cell"><div class="n">32</div><div class="l">一個 warp 中的執行緒數 —— 最小的工作單位</div></div>
</div>

這些數字每一個背後都有故事,而這份指南要講的,就是這些故事。

## 這份指南要怎麼讀

三個部分是層層相扣的設計:

**第一部分 —— 為什麼需要 GPU。** 我們從餐廳開始講起。CPU 像一位才華洋溢的主廚,而 GPU 像一間雇了上千名出餐員的自助餐廳。這部分短小、偏觀念。就算你自認已經知道 GPU 為什麼快,也還是請讀一下 —— 這裡的類比會貫穿整份指南。

**第二部分 —— GPU 架構。** 我們走進廚房參觀。Streaming Multiprocessor、warp、共享記憶體、HBM、Tensor Core。你會看到前面那些數字是從哪裡冒出來的,也會理解為什麼「記憶體階層」比「核心數量」還重要。

**第三部分 —— LLM 推論。** 我們親眼看著一個 token 誕生。Attention、KV 快取、prefill 與 decode、批次處理(batching)。這是 GPU 硬體和 Transformer 軟體交會的地方。你會明白為什麼推論是「受限於記憶體頻寬」的,以及為什麼這短短一句話,幾乎決定了現代推論服務系統的所有設計選擇。

!!! tip "讀完的承諾"
    讀完之後,當有人說「我們在 decode 階段受限於頻寬,需要升級到 H200 才能用更大的 HBM」時,你會清楚地知道他在講什麼 —— 也會大致知道這樣升級能換回多少效能。

## 關於類比的小提醒

類比是鷹架。它幫你爬上牆,但最後你終究會跳下來。這份指南裡的每個類比,都會在某個地方開始失靈,而失靈的那個地方,通常都很有意思。當某個類比不再有幫助的時候,我們會主動告訴你,然後換上一個更精準的。我們的目標是建立那種「跟真實程式碼接觸之後還能存活」的直覺。

<div class="analogy" markdown>
GPU 不是一顆「跑得更快的 CPU」。它是一種完全不同種類的機器,只是剛好共用一個插槽和一套程式語言而已。把它當成「CPU 但更強」來看待,是絕大多數人的程式在 GPU 上跑得慢的頭號原因。
</div>
