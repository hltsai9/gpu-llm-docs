# Capability mapping

「現在的技術能不能做 X?」這是在 LLM 上做產品的團隊最常被問、也最常得到模糊答案的題。這一章是一份地圖。

誠實的答案很少是「可以」或「不可以」。通常是下面三種之一:

- **綠燈。** 技術能可靠做到。做吧。
- **黃燈。** 技術能做到,但要有 guardrail、eval、人類退路。範圍要仔細劃。
- **紅燈。** 技術還做不到。等、或帶著大量 human-in-the-loop 出貨。

大多失敗的 AI 產品,是把黃燈當綠燈出的。

## 這份地圖,以 2025 年而言

以下是對前沿 model 能力的務實讀法。具體數字每次 model release 都會動;類別動得慢一點。

### 綠燈:出貨

- **對已知文字做 summary。** 短 summary、bullet、主管用摘要。前沿各家都可靠。
- **文字重新格式化。** 把純文字轉 Markdown、JSON、bullet list、表格。
- **主要語言之間的翻譯。** 資源最多的大約 50 個語對有接近人類的品質。
- **語氣與風格改寫。** 「讓這段更正式。」「縮成一條 tweet。」
- **從短清單裡分類。** 情感、類別、優先級(落進你定義的桶)。
- **從結構良好的文字裡抽取。** 從 email、合約、文件抽姓名、日期、entity。
- **簡單的程式碼自動補全。** 主流語言的慣用程式碼補全。
- **有界範圍的對話 UI。** 一個有清楚退路、鎖在特定領域的聊天介面。
- **對規格清楚的任務 follow 指令。** 「把這段改寫成 New Yorker 的風格」—— 穩定。

這些可靠到你今天就可以用一般工程水準放到使用者面前,並期待各 model 家族的品質相近。

### 黃燈:有 guardrail 才會動

- **從取回來的 corpus 回答問題(RAG)。** 有好的 retrieval 和拒絕 prompt 就很穩。沒有就當場露餡。見 [RAG](../06-building/rag.md)。
- **多步推理。** 用 chain-of-thought 或 reasoning model,大多「正常」問題能做;真正沒看過的新問題會崩。
- **自動補全以外的程式碼生成。** 給 spec 寫一個小函式 —— 通常對;偶爾細節錯。永遠要 review。
- **創意寫作。** 好的初稿;通常要編輯才能避開模板化的用詞。
- **帶政策合規的複雜客服。** Model 能跟政策;你要能證明它跟了。人類 QA 抽樣是必要的。
- **範圍窄的 agent。** 見 [agent](../06-building/tool-use-agents.md)。特定、有邊界的 workflow 能動;開放式的會壞。
- **用使用者自己的聲音改寫他的 email。** 有範例就能動;仍可能 drift。
- **從非結構化輸入做表格推理。** 「從這份 PDF 抽發票總額」—— 通常能;偶爾看錯。
- **按明確規則做 moderation。** 政策清楚就能做;尾端 case 要人。

每一個黃燈功能都需要:清楚的拒絕路徑、量 drift 的 eval、以及出事時的 production 計畫。

### 紅燈:還不夠可靠

- **特殊領域的事實 recall。** 醫療、法律、監管。model 很有把握,而且常常錯。永遠要求從權威 source 做 retrieval,不要靠訓練資料的 recall。
- **Agent 自主採取不可逆行動。** 付款、寄 email 給外部、修改客戶資料而沒人類核准。錯誤率沒低到足夠,而錯誤成本太高。
- **長時程 planning。** 跨越幾小時或幾天的多步任務。現在的 agent 會失去連貫;短跑可以,馬拉松不行。
- **超越國小程度的數學。** 帶 reasoning mode 的 model 大幅改善,但非瑣碎問題仍會漏。用計算器或 code interpreter tool。
- **任何需要完美可靠的事情。** 醫療診斷、法律意見、金融交易、實體世界控制。不是因為 model 永遠錯,而是偶發的「有信心地錯」成本無法接受。
- **前沿性研究。** Model 在重組;它不會在前沿可靠地發明。
- **沒有工程補強的對抗使用者防禦。** Prompt injection 是未解問題;任何有敵意的人,基本上都能繞過基礎的 prompt 防禦。
- **真正個人化的長期記憶。** 現在的「記憶」功能是對過去對話的 retrieval,不是真正持續的學習。有幫助,但不如人類記憶。

這些要嘛是現役研究方向,要嘛是結構性限制。做一個依賴它們的產品,就是在賭還沒出貨的東西。

## 領域別的誠實觀察

有些類別值得近看,因為平均值藏了變異。

### 寫程式

- **自動補全、樣板、小重構:** 綠。
- **給測試和 spec 寫新函式:** 黃。
- **理解一份你不熟的大型 codebase:** 黃,加好的 context 工具。
- **Agentic coding(plan + 實作 + test):** 範圍清楚的任務是黃;「build this app」這種開放式的是紅。
- **安全關鍵程式碼:** 沒專家 review 上 production 是紅。

### 客服

- **Tier-1 FAQ + RAG + 拒絕:** 綠。
- **問題分類與路由:** 綠。
- **政策敏感互動(退款、取消):** 把政策放進 context 且有人工 review 是黃。
- **帳務或帳號行動:** 每一個行動都要人明確核准才算黃。

### 內容創作

- **初稿:** 綠。
- **帶好範例的品牌聲音初稿:** 黃。
- **不像模板的長文:** 黃(需要大量編輯)。
- **事實報導:** 沒人類 fact-check 是紅。

### 資料與分析

- **自然語言轉 SQL:** 黃。簡單 schema 能動;複雜 join 會壞。執行前一定先 preview。
- **把結構化資料 summary 成敘事:** 綠。
- **偵測數值資料的異常:** 紅(用真的 anomaly detection;LLM 不是對的工具)。

### 醫療 / 法律 / 金融

- **草擬型工具(memo、case note、template):** 黃,要專家 review。
- **在內部政策 / 文件裡搜尋:** 黃,搭 RAG。
- **對終端使用者的建議:** 大多管轄區和 use case 都是紅。

## 怎麼校準

公開 benchmark 有幫助但不拍板。Benchmark 只在「你的產品長得像 benchmark」時能預測你的效能。

**每一個要範圍化的功能,建一組 50 個範例的 eval 集**(見 [eval](../06-building/evals.md)),跑 2–3 個 model 候選。這比任何排行榜更能預測。

被迫在 build eval 之前就決定時:

- **讀最近的 model card。** 不是行銷文 —— 是*limitations*那節。
- **問工程師失敗模式是怎麼露出來的。** 「它錯的時候長什麼樣?」比「它多常對?」更有資訊。
- **看類似已出貨產品。** 如果一個資金充足的團隊試過 X、又靜靜收掉,那就是訊號。

## 速度問題

能力每幾個月就動。今天是黃,半年後可能變綠。怎麼規劃:

- **產品架構圍繞黃燈 baseline 建**(guardrail、eval、人類退路),這樣可靠度成長了你再放鬆。不要把 roadmap 押在「它會變好」上。
- **每次重大 model release 後重跑 eval。** 有時一夜之間分數動 10 點。
- **Prompt 和 tool 要穩。** 脆弱的通常不是 prompt,是 model;model 變強的時候,你現有的程式碼常常就跟著變好。

## 一份可用的功能 scoping 模板

任何被提案的 LLM 功能,在寫 spec 之前,先為下面每一項寫一段:

- **這在 capability map 上是哪種顏色?** 綠 / 黃 / 紅,附理由。
- **失敗成本是什麼?** 如果 5% 的 case 錯,會怎樣?可逆嗎?尷尬嗎?涉法嗎?
- **per-request 經濟學是什麼?** 見[經濟學](economics.md)。
- **需要哪些 guardrail?** 拒絕、人類 review、rate limit、結構化輸出。
- **Eval 是什麼?** 「好到可以出貨」用什麼數字表示?

任何一段是手一揮帶過的,這功能就還沒到能丟給工程的狀態。

## 常見錯誤

- **從一個漂亮 demo 做概括。** Demo 是最佳 case;production 流量是分布。
- **把「流暢」當成「正確」。** 流暢是免費的;正確才是要做工的。
- **把 benchmark 當排名。** 它們是過濾器,不是預測。
- **押在紅燈能力,指望它變好。** 有時會;常常不在你時程上。
- **Model 更新之後不回頭重看這份地圖。** 半年前的功能 spec 就是過時的。

## 一句話總結

Capability map 有三種顏色,不是兩種 —— 綠上面建,黃上面小心,紅不要拿來押產品,除非現實追上了 pitch。
