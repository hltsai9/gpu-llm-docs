# 自助餐廳 vs. 牛排館

想像街道兩側相對開著兩家餐廳。

**牛排館**雇了八位廚師。每一位都是身經百戰的老手。任何一位都可以接單、走進食材冷藏室挑材料、開始熬醬、試味、調整、把菲力煎到完美、擺盤、點綴、送上餐桌。他們無時無刻都在做決策:*「這塊肉看起來厚一點,我多煎兩分鐘」*、*「那一桌看起來趕時間,我先出他們的配菜」*。每一位廚師從頭到尾負責不同的訂單。訂單複雜、可以客製化、充滿不確定性。出餐慢,但每一道菜都正是客人點的樣子。

街對面的**自助餐廳**只雇了一位主廚和一千位出餐員。主廚大聲下指令:*「所有人,切一顆洋蔥。」* 一千把刀同時落下。*「所有人,把洋蔥下鍋。」* 一千口平底鍋同時滋滋作響。這些出餐員個別來看都不是什麼天才 —— 大部分人如果沒有人一步一步教,連個三明治都做不出來。但只要在一聲同步指令下合作,他們可以在牛排館出完前八張單之前,先餵飽整座體育場。

<div class="analogy" markdown>
CPU 是牛排館,GPU 是自助餐廳。
</div>

所有 GPU 的設計選擇,都是從這個定位出發;它們之所以對 LLM 來說像魔法,而對你筆電上的程式卻只是「還可以」,原因也都從這裡開始。

## 牛排館在優化什麼

CPU 被打造來**把「無法預測的連續工作」處理得很好**。它的設計優先順序是:

- **單一任務的低延遲。** 任何一道指令都要盡可能快地完成。
- **分支與決策。** 程式碼整天都在寫 `if user.is_admin: ... else: ...`。CPU 有一大堆龐大的電路(分支預測器、亂序執行引擎)專門負責「猜」下一道指令會是什麼,這樣它就不用停下來等。
- **大型、聰明的快取。** CPU 的 L1/L2/L3 快取就是它的「mise en place(備料台)」—— 如果廚師剛才碰過某樣食材,下次它就已經在眼前。就算沒碰過,CPU 也有一堆邏輯在猜接下來會用到什麼。
- **核心少,但每個都很強。** 現代伺服器 CPU 大概有 32 到 128 個核心。每一個都像一座獨立的矽晶片城市,專心把一條執行緒跑到極致。

CPU 的大部分電晶體預算,都花在讓「單一執行緒」跑得越快越好、越少停頓越好。這也是為什麼 CPU 在資料庫、作業系統、編譯器、試算表這類「分支多、一次服務一個請求」的工作上無可匹敵。

## 自助餐廳在優化什麼

GPU 是為了**大量、可以預測、而且長得一模一樣的工作**而生的。它的設計優先順序是:

- **吞吐量勝過延遲。** 單一一次運算慢沒關係,只要整體每秒能完成「幾百萬次」就好。
- **最少的分支。** 絕大多數核心都在對不同的資料執行完全相同的指令。它幾乎沒有「猜下一步」的機制 —— 大家就是照主廚說的做。
- **每個核心的快取很小,但記憶體管道超粗。** 每條執行緒的聰明程度降低,但整體原始頻寬拉到最高。
- **上千個簡單的核心。** H100 上有大約 16,000 個 CUDA 核心。每一個都不怎麼聰明,但數量夠多,而且它們同步前進。

H100 的電晶體預算和 CPU 是反過來配置的。大部分矽晶片都拿去做算術單元 —— 乘法器和加法器 —— 只留下剛好夠用的控制邏輯,把它們餵飽就行了。

## 為什麼 LLM 愛自助餐廳

接下來是關鍵。Transformer 在推論時到底在做什麼?對每一層、每一個 token,它的核心工作都是一個巨大的矩陣乘法:

$$
Y = X W
$$

其中 $X$ 和 $W$ 都是有著上千列、上千行的矩陣。兩個 4096×4096 矩陣相乘一次,大約需要 **1370 億次**乘加運算。而且這些運算幾乎完全彼此獨立:計算輸出格 $(i, j)$ 不會依賴計算 $(i, j+1)$。你可以把每一格交給不同的工人去算。

這「剛剛好」就是自助餐廳被設計來處理的那種形狀的工作。「各位,把你被分配到的 X 的那一列,乘上你被分配到的 W 的那一行。開始。」一萬個核心、一萬個獨立的小點積,沒有分支、沒有戲劇性。

<div class="analogy" markdown>
叫 CPU 做 70B 參數的矩陣乘法,等於是請八位牛排館廚師一次做一個漢堡、一次做一萬個一模一樣的漢堡,每個漢堡都還要停下來想想「這顆需不需要多撒一點鹽」。叫 GPU 做同樣的事,則是對整間自助餐廳喊一聲:*「各位,翻面。」*
</div>

## 類比會在哪裡失靈(為什麼知道這個很重要)

每個類比都有邊界。以下是這個類比的幾條邊界,因為它們在後面章節會派上用場。

**出餐員不是各自獨立運作的。** 真實自助餐廳裡,如果有兩位出餐員比較慢,主廚就繞過他們自己調整。但在 GPU 上,一千位出餐員是分成每 32 人一組、叫做 **warp** 的編制在走,而他們必須「在同一個時間、執行同一道指令」。如果其中一位出餐員因為 `if` 分支要跑不一樣的動作,剩下 31 位就只能站在那裡閒等。這個我們會在 [SIMT](simt.md) 那一章細講,它是「天真寫出來的 GPU 程式跑得慢」的頭號原因。

**倉庫沒那麼近。** 在廚房裡你會想像食材伸手就能拿到。但在 GPU 上,主記憶體(叫做 **HBM**,high-bandwidth memory 的縮寫)以電腦的標準來說很快,但跟核心做乘法的速度比起來卻「很慢」。寫出好的 GPU 程式有一大部分,就是在避免一直跑回倉庫拿東西。這部分我們在 [記憶體階層](../02-gpu-architecture/memory-hierarchy.md) 會仔細看。

**主廚不只是在吆喝而已。** 真實的 GPU kernel 底下隨時在做排程、同步、搬移記憶體。所謂「一道指令」的模型,是一個乾淨的第一印象,不是全貌。

不過還是把自助餐廳的畫面留在腦中。等你之後讀到「warp divergence 很貴」或「HBM 頻寬是瓶頸」的時候,這個畫面會直接告訴你原因 —— 要嘛是出餐員站在那裡閒等,要嘛是食材還在從倉庫被推車運過來。

## 一句話總結

CPU 是被設計來快速完成一個困難任務的。GPU 是被設計來「同時」完成一整座山那麼多簡單、重複的任務的。而 LLM,就是一整座山的簡單、重複任務。
