# 同步運作的廚房(SIMT)

到目前為止,我們的自助餐廳只有一位主廚對一千位出餐員吆喝指令。這離真相很接近,但還差一點。真實的樣貌更具體 —— 而那些具體的細節,正是效能從哪裡贏、從哪裡輸的地方。

GPU 的真正規則是這樣:

<div class="analogy" markdown>
出餐員是以 **32 人一組** 的編制在工作。編制裡的每一位出餐員必須 **在同一時刻、執行同一個指令**,只是作用在 **不同的食材** 上。如果有一位出餐員要做不一樣的動作,剩下 31 位就只能站在那裡握著刀等。
</div>

這個 32 人編制叫做 **warp**。程式設計模型叫做 **SIMT** —— Single Instruction, Multiple Threads(單指令、多執行緒)。這是你最需要內化的 GPU 觀念。

## 為什麼是 32,為什麼要同步?

記住:GPU 用「犧牲每條執行緒的聰明度」換來「平行吞吐量」。其中最大的一筆節省來自**指令解碼器**。CPU 每個核心都有一個複雜的解碼器,消化複雜指令、預測分支、即時重新排序執行順序。這塊矽晶片很貴。

GPU 把解碼器「共用」給 32 條執行緒。一道指令抓一次、解一次,然後同時套用到 32 筆資料上。同樣的工作量,解碼器的電路面積少了 32 倍 —— 這就是為什麼它能在一顆晶片上塞 10,000 個核心。

代價就是同步規則。一個 warp 裡的 32 條執行緒共用同一個指令指標(instruction pointer)。它們可以處理不同的資料,但必須執行同一道運算。

!!! info "SIMT vs SIMD"
    CPU 有一個類似的概念叫做 SIMD —— 單指令、多資料,透過像 AVX-512 這種向量暫存器來實現。差別在哪?SIMD 是「指令層級」的功能(單一核心,但指令操作一次處理一整排向量)。SIMT 是「執行緒層級」的功能(一大堆執行緒假裝彼此獨立,但硬體在底下偷偷強迫它們同步)。SIMT 比較好寫,因為你大部分時候可以假裝每條執行緒都獨立,直到你撞上 divergence(分歧)。

## 分歧(Divergence)問題

當你的程式裡出現 `if` 會怎樣?

```python
# 在 GPU 上,32 條執行緒每一條都會跑這段:
if value > 0:
    result = sqrt(value)
else:
    result = -1.0
```

既然 32 條執行緒必須同時執行同一道指令,硬體沒得選。它會先替條件為真的那些執行緒跑 `if` 分支 —— 其他執行緒**站在那裡閒等**,暫時被遮罩掉。然後再替剩下的執行緒跑 `else` 分支,這時換第一組站在那裡看。兩個分支會依序執行,而在任何一個瞬間,都只有部分執行緒在做有意義的事。

這叫做 **warp divergence(warp 分歧)**。最糟的情況下,如果 32 條執行緒每一條都走不一樣的路徑,你會損失到 32 倍的吞吐量。

<div class="analogy" markdown>
想像主廚大喊:*「如果你手上是紅蘿蔔,就削皮;不是就切。」* 拿到紅蘿蔔的人開始削皮,負責切的人站在一旁握著刀看。紅蘿蔔削完了,換負責切的去切,削皮的站在一旁看。兩倍的時間,同樣的總工作量。
</div>

對 LLM 來說,這大多不是什麼問題 —— 矩陣乘法裡沒有分支。每條執行緒都在對不同的數字做同一個 `乘加` 運算。Transformer 幾乎是完美均勻的工作,這也正是 GPU 服務它們服務得這麼好的原因。

## 為什麼 LLM 的數學和 SIMT 模型完美契合

一次矩陣乘加就是:

```
accumulator += A[i,k] * B[k,j]
```

對於輸出矩陣裡的每一個 (i, j),這道指令都是「一模一樣」的。你可以把每個 (i, j) 交給一條執行緒,它們會對不同的 A 和 B 元素執行同一個運算。沒有分支、沒有分歧。每個 warp 裡的每位出餐員都在做同一件事 —— 只是各自處理矩陣中被分配到的那一小塊。

這就是為什麼矩陣乘法在 GPU 上常常比在 CPU 上快 100 到 1000 倍。它是這塊硬體被打造來吃的「運算形狀」。Transformer 的 attention、前饋層(feedforward)、layer normalization —— 現代 LLM 裡幾乎所有東西,到底層都歸結為矩陣乘法加上一些 elementwise 運算。全部都是 warp 友善的。

<div class="analogy" markdown>
對自助餐廳喊一句 *「所有人,切丁你的洋蔥」* 是全世界最簡單的指令。而 Transformer 從頭到底,本質上就是這樣。
</div>

## 工程師要小心的地方

雖然 Transformer 整體是 warp 友善的,真實的 GPU kernel 還是得處理下列這些會造成分歧的情境:

- **帶遮罩的 attention。** 在因果 self-attention 裡,某些位置不應該看到未來的 token。這個遮罩會帶來 `if`,可能造成分歧。
- **稀疏運算。** 稀疏矩陣程式(例如 MoE 的路由)會讓不同執行緒走向不同的專家模型。
- **邊界條件。** 當矩陣維度不是 32 的倍數時,最後一個 warp 裡有一部分執行緒在做事,另一部分沒事做。

寫得好的 GPU kernel,會把工作編排成「一個 warp 裡的 32 條執行緒都走同一條路徑」。寫得不好的不會,代價就是閒置核心。編譯器可以幫一些忙(predication、warp-level primitives),但問題本身的「形狀」比編譯器的聰明度重要得多。

## 思維上的切換

到這裡,你已經有足以理解「LLM 如何跑在 GPU 上」的畫面了:

- 數千個極小的核心。
- 以 32 人一組的 warp 為單位。
- 每個 warp 一次執行一道指令,完美同步。
- 整顆晶片靠不斷切換許多 warp 來讓自己保持忙碌;只要某個 warp 卡在記憶體讀取,就換別的 warp 進來。
- LLM 的數學天生就是「無分支、均勻、龐大」—— 正是這台機器被打造來處理的那種工作負載。
