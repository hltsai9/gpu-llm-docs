# 吞吐量優於延遲

評估一間廚房,有兩種方式。

**延遲(Latency)**是從你下單開始,到那一道「你的餐」端上桌為止的時間。當你肚子餓的時候,你在乎的是這個。

**吞吐量(Throughput)**是這間廚房每小時出多少道餐。當你是餐廳老闆的時候,你在乎的是這個。

CPU 是為「你那道菜」優化的。GPU 是為「餐廳的星期五晚上」優化的。

## 高速公路的類比

換個場景想一下。假設你要把一萬個人從舊金山送到洛杉磯,有兩種策略。

**跑車策略。** 一台法拉利。它跑完 380 英里需要三個半小時。然後調頭開回去,接下一位乘客,再跑一趟。法拉利在「單趟上」很快 —— 每位乘客的「從門到門延遲」大概 7 小時(含往返)。要載完一萬人,它需要跑一萬趟。總時間:好幾年。

**貨運列車策略。** 一列火車。它開到 LA 要 8 小時(每位乘客反而比較慢!)。但它一次載一萬人。總時間:8 小時。

法拉利贏在「每位乘客的延遲」。火車贏在「乘客的吞吐量」。只要是要搬運一群人,火車會以天差地遠的差距獲勝,儘管它單趟跑得比較慢。

GPU 就是那列火車。

<div class="analogy" markdown>
GPU 上的單一運算,常常比 CPU 上「同一個運算」還要慢。GPU 之所以贏,是因為它在 CPU 做完第一個運算的時間裡,已經做完了一百萬個。
</div>

## 為什麼電晶體要這樣分配

晶片設計師有一個固定的電晶體預算。他們可以把預算花在「讓單一執行緒跑得更快」,或「同時平行跑更多執行緒」上。兩個不能兼得 —— 矽晶片就這麼大,而這兩個目標方向剛好相反。

CPU 把預算花在:

- 分支預測器,用來猜 `if` 會走哪一邊。
- 亂序執行(Out-of-order execution),這樣某一道指令卡住時,核心還能先處理後面的指令。
- 巨大的快取,讓下一個要用的 byte 永遠只差幾奈秒。
- 寬闊的解碼器,用來消化複雜的 x86 指令。

這些機制全都是為了從「一條執行緒」上省下奈秒等級的時間。它們對「平行跑更多執行緒」毫無直接幫助。

GPU 把預算花在:

- 更多的算術單元(乘加器)。
- 更多同時在飛的執行緒(單一 H100 的 SM 可以同時處理到 2,048 條執行緒)。
- 寬闊的記憶體通道,用來餵飽所有這些單元。

GPU 幾乎不想把單一一條執行緒弄快。它要做的,是確保在晶片的某個角落「隨時」都有算術在發生。如果第 47 條執行緒卡住在等記憶體,第 48 到 127 條會繼續前進。從外面看,根本感覺不到那次卡住,因為總是有人在工作。

這叫做**延遲隱藏(latency hiding)**,是 GPU 效能背後最重要的一招。

## 一張圖看懂延遲隱藏

想像一位出餐員在等一鍋水煮開。在 CPU 上,這位廚師就站在那裡看著鍋子。在 GPU 上,當一位出餐員在等水開時,已經有另外 2,000 位出餐員在切、在炒、在擺盤 —— 從主廚的角度看,整間廚房從來沒有慢下來過。

!!! info "用 GPU 術語來說"
    當一個 warp 發出向 HBM 讀取記憶體的請求時,它可能得等幾百個 cycle 資料才會到。SM 不會讓它閒著,而是立刻切換到另一個已經能開始運算的 warp。等原本那個 warp 的資料到貨時,再把它換回來。晶片上的算術單元幾乎可以保持接近 100% 的忙碌 —— 但「每一條執行緒」的延遲卻可以大得嚇人。

這也是為什麼在 GPU 上「吞吐量」是唯一該在乎的數字。單獨看每一條執行緒都很慢,但全部加起來就很兇猛。

## 這對 LLM 的意義

有兩個實際的後果,值得把它們刺在顯眼的地方:

**1. 小 batch 是浪費 GPU。** 如果你一次只送一個請求給 GPU,那就是派法拉利去載一位乘客。整列火車都空著一半以上。LLM 服務系統會拼命把很多請求「批次」起來,讓 GPU 一次大矩陣乘法就同時替 64、128,甚至 256 場對話服務。這部分怎麼運作,我們在 [同時服務多桌客人(Batching)](../03-llm-inference/batching.md) 會講。

**2.「每個 token 的延遲」和「token 的吞吐量」不是同一件事。** 一套「一個使用者每秒 100 token」的系統,和一套「跨 100 個使用者每秒 10,000 token」的系統,是完全不同的東西。前者在優化延遲,後者在優化吞吐量。LLM 基礎架構團隊必須先選定目標,再針對它設計 —— 你沒辦法把兩者同時拉到極致。

<div class="analogy" markdown>
一間咖啡店可以走「每位客人很快」(一位咖啡師專心做一杯),或「總量爆高」(像流水線一樣每人專責一個步驟),但在極限狀態下兩者不能兼得。LLM 服務面對的是同樣的取捨,只是每次取捨的代價,要乘上一張 GPU 的價錢。
</div>

## 思維上的切換

當你發現自己在想「這單一個運算有多快?」時,停下來。改問:「我一次能平行跑多少個?晶片有沒有被我填滿?」從延遲思維轉成吞吐量思維,這是真正「GPU 思維」的第一步。
