# GPU 的解剖學

讓我們去工廠現場走一圈。

一顆現代的資料中心 GPU —— 例如 NVIDIA H100 —— 其實就是一塊差不多郵票大小的矽晶片,上面塞了 800 億顆電晶體。如果我們把鏡頭拉近,會發現它「不是」一個巨大的機器。它是一個由**許多較小、彼此完全相同的機器**組成的社區,每一個叫做 **Streaming Multiprocessor(SM)**。H100 上面有 132 個。

<div class="analogy" markdown>
把 GPU 想成一座工廠園區,裡面有 132 棟一模一樣的工作坊。每一棟就是一個 SM。每間工作坊裡有工作台、工具、小型倉庫,還有一組工人。工作坊外觀都一樣,做的也都是一樣的工作。園區的使命是:讓每一棟工作坊都保持忙碌。
</div>

## 一個 SM 裡面住著什麼

打開某一棟 SM 的門,大致會看到:

- **大約 128 個 CUDA 核心。** 這些就是出餐員。每一個都是簡單的算術單元,可以一個 cycle 對 32 位元浮點數做一次乘加運算。不聰明,但速度快,而且每個 SM 裡有一堆。
- **4 個 Tensor Core。** 矩陣乘法的專業級電動工具。一個 Tensor Core 可以一步完成一個小矩陣乘法(例如 4×4 × 4×4),而不是分成很多步。在 LLM 數學裡,大部分的真實算術其實是在這裡發生的。[細節看這裡](tensor-cores.md)。
- **Warp 排程器(通常 4 個)。** 領班。每個 warp 排程器每個 cycle 可以對一個 warp(32 條執行緒)下達一道指令。有 4 個排程器,一個 SM 就能同時讓 4 個 warp 每 cycle 都在執行。
- **暫存器檔(Register file)。** 工作台上的那排工具。每個 SM 大約有 64K 個 32 位元的暫存器 —— 分給所有活躍的執行緒共用。執行緒在跑的時候,變數就活在這裡。這是晶片上最快的儲存。
- **共享記憶體 / L1 快取。** 工作坊自己的小倉庫。每個 SM 大約 256 KB,一個 block 內的所有執行緒可以共用。比主記憶體快好幾個數量級。
- **載入 / 儲存單元、特殊函式單元。** 搬資料進出的管線,以及算超越函數(sin、cos、exp 等)的專用電路。

把這一切乘上 H100 上的 132 棟工作坊,你就得到:

<div class="cell"><div class="n">528 個</div><div class="l">Tensor Core 總數</div></div>

## 控制塔與倉儲

包圍著這 132 棟工作坊的,還有兩個「整顆晶片」共享的資源:

**L2 快取。** 一塊大的、橫跨整顆晶片的快取(H100 大約有 50 MB),坐在 SM 與主記憶體之間。你可以把它想成工廠園區裡的**中央派送倉庫** —— 如果剛才有任何一棟 SM 碰過某筆資料,它大概就躺在這裡,其他 SM 也可以快速拿到。

**HBM(High Bandwidth Memory,高頻寬記憶體)。** 主記憶體。H100 上這是 80 GB 的特製 DRAM,堆疊在「緊鄰」GPU 晶片的位置,透過一條 5,120 位元寬、跑在約 3 TB/s 的記憶體匯流排連接。HBM 就是讓 GPU 在「吃記憶體」的工作上能跑這麼快的原因 —— 它在整個運算界裡,幾乎就是最寬的記憶體介面之一。

<div class="analogy" markdown>
如果暫存器檔是你的工作台、共享記憶體是工作坊的小倉庫,那 HBM 就是一個巨大的裝卸月台,堆高機一次把一整板原料送進來。以一般標準來看它很快 —— 但它比小倉庫遠,每來回一趟都得花時間。
</div>

關於「距離」以及它帶來的後果,我們在 [記憶體階層](memory-hierarchy.md) 會深入談。現在只要把這個形狀吸收進去就好:**快速、小容量的儲存靠近核心;慢速、大容量的儲存在遠方。**

## GigaThread 引擎(輪班經理)

誰決定哪一棟工作坊做哪一份工作?**GigaThread 引擎**是整顆晶片的全域排程器。當你啟動一個 kernel(一個 GPU 函式),例如切成 1,024 個 thread block,GigaThread 引擎會把這些 block 分派給 132 個 SM。每當一個 SM 做完手上的 block,引擎就再塞下一個給它。

寫程式的人「不會」挑哪個 SM 跑哪個 block。你只說:「我要把這個函式跑在一個 1D/2D/3D 的 thread block grid 上」,剩下的硬體自己決定。這種「寫程式的人不用管」的脫鉤,也是 GPU 程式能自動擴充的原因之一 —— 一樣的程式可以在只有 20 個 SM 的消費級卡上跑,也能在 132 個 SM 的資料中心卡上跑,只差在快慢,不用改一行原始碼。

## 這個規模對 LLM 意味著什麼

當你在 H100 上跑一次 Transformer 的前向傳遞時,模型的 70B 參數是住在 HBM 裡(假設是 FP16,大約 140 GB —— 超過單卡容量,這部分後面再講)。要把一個權重矩陣乘上一個 activation:

1. 相關的權重 tile 必須從 HBM 搬到每個 SM 的共享記憶體 / 暫存器裡。
2. Tensor Core 在暫存器上執行矩陣乘法。
3. 結果再搬回 HBM(如果是中間結果就繼續留在晶片上)。

瓶頸幾乎從來不是 Tensor Core。它們快到 HBM 根本餵不上。整個有效率地推論 LLM 的藝術,就是**如何讓 Tensor Core 不餓肚子** —— 也就是聰明地決定要搬哪些資料、什麼時候搬、搬過來之後在晶片上還能重複用多少次,才在它被逐出之前完成工作。

這正是 [記憶體階層](memory-hierarchy.md) 要鋪陳的內容,你接下來就該讀那一章。

Tensor Core 是過於優秀的電動工具。它們完工的速度比裝卸月台把原料送到還快。整場遊戲的重點,就是不要讓它們閒下來。
</div>
