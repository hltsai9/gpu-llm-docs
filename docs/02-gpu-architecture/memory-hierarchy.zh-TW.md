# 記憶體階層

這整份指南如果你只讀一章,就讀這一章。

幾乎每一場關於 LLM 效能的真實討論 —— 量化、FlashAttention、paged KV cache、模型切片、H100 vs H200、為什麼推論延遲會長成那個樣子 —— 到頭來都是在討論記憶體。不是核心。核心沒問題。核心其實「過於」優秀。整場遊戲是:怎麼把資料及時送到核心手上。

## 一座儲存的城市

這一章換個類比。把每條執行緒想成一位需要食材的廚師。食材住在不同距離的地方:

| 儲存 | 位置 | 容量 | 速度 | 誰看得到 |
|---|---|---|---|---|
| **暫存器(Registers)** | 廚師手上 | 每個 SM 約 256 KB(每條執行緒分到一點) | 1 個 cycle | 單條執行緒 |
| **共享記憶體 / L1** | 工作坊的架上 | 每個 SM 約 256 KB | 約 30 個 cycle | 一個 block 內所有執行緒 |
| **L2 快取** | 園區中央倉儲 | 整顆晶片約 50 MB | 約 200 個 cycle | 所有 SM |
| **HBM(主記憶體)** | 停車場對面的裝卸月台 | 80 GB | 約 500–800 個 cycle | 整顆 GPU |
| **透過 PCIe 到 CPU 的記憶體** | 另一座城市 | 數百 GB | 約幾萬個 cycle | 主機端 + GPU(靠複製) |
| **NVMe / 硬碟** | 另一個國家 | TB 等級以上 | 百萬個 cycle | 只有主機端 |

<div class="analogy" markdown>
暫存器是你手上拿著的東西。共享記憶體是背後那排架子。L2 是同棟建築另一邊的倉庫。HBM 是隔壁停車場的裝卸月台。CPU 記憶體是另一頭的供應商。硬碟是海外。
</div>

速度差可以跨到好幾個數量級 —— 從暫存器到硬碟的比例大約是 **十億比一**。單憑這個事實,就決定了 GPU 軟體裡幾乎每一個設計選擇。

## 頻寬 vs. 延遲,再談一次

每一個階層都有兩個要記住的數字:

- **延遲(Latency)** —— 從這裡拿「一個 byte」要花多久。
- **頻寬(Bandwidth)** —— 如果你一次請求一大堆,每秒能搬多少 byte。

HBM 就是很好的例子。它的延遲「超爛」—— 好幾百個 cycle。但它的頻寬是怪物等級的 —— H100 上 3 TB/s。這也是為什麼 GPU 喜歡把資料「大塊、連續地」搬:延遲只付一次,之後整段搬運都坐在頻寬上順風車。

<div class="analogy" markdown>
開往裝卸月台的貨運列車啟動要花很久時間。但它一旦上路,載貨量大得驚人。一次只拿一箱是浪費;整列拉滿,才是系統當初被設計出來的使用方式。
</div>

用 GPU 的術語說,這叫做 **合併式記憶體存取(coalesced memory access)**:一個 warp 裡的 32 條執行緒應該載入 32 個「相鄰」的 byte,這樣硬體一次寬 HBM transaction 就能滿足所有執行緒。如果執行緒跳來跳去存取隨機位址,每條執行緒都要付完整延遲,吞吐量就崩掉了。

## 重複使用:聖杯

這是 LLM 推論的核心洞察:

<div class="analogy" markdown>
每一個你可以在送回裝卸月台前「用兩次」的 byte,就是一個「不用再抓一次」的 byte。
</div>

假設你有一個 4096×4096 浮點數的權重矩陣 $W$(FP16 下是 64 MB)。你要把它乘上一個長度 4096 的向量 $x$。這個矩陣乘法會碰到:

- 64 MB 的 $W$(讀一次)
- 8 KB 的 $x$(讀一次)
- 8 KB 的結果 $y$(寫一次)

總記憶體流量:約 64 MB。總算術量:約 3300 萬次乘加。

現在把它改成「批次」乘 64 條向量 —— 把單一向量換成一個 4096×64 的矩陣 $X$。這個矩陣乘法會碰到:

- 64 MB 的 $W$(還是只讀一次 —— 我們在 batch 裡重複使用它!)
- 512 KB 的 $X$
- 512 KB 的 $Y$
- 算術量:約 21 億次乘加 —— **64 倍的工作量**

只多了一點點記憶體流量(多 1 MB 左右),我們多做了 64 倍的計算。**每搬一個 byte 能做多少運算** 大幅提升。這個比例 —— **算術密度(arithmetic intensity)** —— 就是神奇數字。

這也是「為什麼 batching 對 LLM 這麼重要」的全部原因。當你把許多請求一起送過同一個權重矩陣時,HBM 的代價只付「一次」,然後把它平均分攤到所有請求上。我們會在 [Batching](../03-llm-inference/batching.md) 和 [Roofline](../03-llm-inference/roofline.md) 章節再遇到這件事。

## 記憶體牆(The Memory Wall)

下面這件事對現代 GPU 來說很不好受:

<div class="numbers" markdown>
<div class="cell"><div class="n">989 TFLOPs</div><div class="l">H100 的 BF16 算力</div></div>
<div class="cell"><div class="n">3.35 TB/s</div><div class="l">HBM 頻寬</div></div>
<div class="cell"><div class="n">約 295</div><div class="l">平衡點:每 byte 要做多少次運算</div></div>
</div>

要在 H100 上達到「打平」—— 也就是只靠 HBM 就能把 Tensor Core 完全餵飽 —— 每一個從 HBM 載入的 byte,在被丟棄前都得參與大約 295 次算術運算。如果你只讀一個 byte 卻只對它做 10 次運算,Tensor Core 會在 HBM 送到這個 byte 所需時間的一小部分就完工,然後剩下時間都在閒置。

這就是**記憶體牆**。核心在過去十年裡變得快非常多;HBM 頻寬也有成長,但沒那麼快。兩者差距越拉越大。寫 GPU 程式的藝術,越來越變成「如何不去 HBM」的藝術。

## LLM 裡每一層記憶體都怎麼被使用

跟著 Transformer 的一層走一遍,看資料是怎麼流動的。

**權重**住在 HBM。它們太大,哪裡都塞不下 —— 單一 70B 參數的模型在 FP16 下是 140 GB,要嘛切分到多顆 GPU 上,要嘛量化。當你做一次矩陣乘法時,權重矩陣的 tile 會從 HBM 串流到晶片上的記憶體。

**權重和 activation 的 tile** 會被暫存在每個 SM 內部的**共享記憶體**裡。多個 warp 反覆重用這些 tile,在 tile 被逐出前做很多次乘加。這也是「算術密度」實際發生的地方 —— 一次 HBM 載入、很多次運算。

**矩陣乘法最內層的累加器**住在**暫存器**。Tensor Core 從暫存器讀運算元、把結果寫回暫存器。

**L2 快取**在多個 SM 剛好碰到重疊資料時幫忙 —— 例如不同的輸出 tile 使用到同一個權重矩陣。一個好的 tile 切法會刻意安排出這種重用。

**推論時的 KV 快取**也是住在 HBM。在 decode 階段,每產生一個新 token 都得抓整份 KV 快取來做 attention —— 而這次抓取,正是 decode 被記憶體頻寬卡住的原因。[詳見 KV 快取章節](../03-llm-inference/kv-cache.md)。

## 為什麼「量化」其實就是「縮小那趟去裝卸月台的搬運量」

現代 LLM 服務到處都在談量化:把模型從 FP16/BF16 改跑 INT8、FP8 或 INT4。表面上的好處是模型占用記憶體變小。**更大**的好處是,每一次 HBM 讀取搬的 byte 量減半甚至變四分之一。既然 decode 是被頻寬卡住的,把 HBM 流量減半,每秒輸出的 token 大致就能翻倍。

<div class="analogy" markdown>
量化不是把廚師變快。它是把每個食材都裝成更小的箱子,讓堆高機一趟能送兩倍的量。
</div>

## 值得帶走的重點

- 記憶體階層在容量和速度上橫跨大約 **九個數量級**。
- GPU 的獲勝方式,是盡可能讓核心從**晶片上的記憶體**(暫存器、共享記憶體)被餵飽。
- 要持續維持接近巔峰算力的唯一方法,就是保持**高算術密度**:每搬一個 byte,要做很多次運算。
- Batching、量化、FlashAttention、paged KV cache、operator fusion —— 它們其實都是同一句話的不同說法:*「少搬資料、搬過來的重複使用。」*

一旦你腦中有這幅畫面,現代 LLM 服務系統裡那些看起來很聰明的演算法,都會變得好懂許多。它們的重點不是從核心身上再榨出更多,而是從每一個跨越記憶體階層的 byte 身上,再榨出更多工作。
